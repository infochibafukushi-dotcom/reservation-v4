# 現状分析レポート（reservation-v4）

## 1. 全体構成
- フロントエンドは静的HTML + Vanilla JS（`index.html`, `form.html`, `admin.html`）で構成。
- バックエンドはCloudflare Workers想定の単一ファイルAPI（`worker.js`）。
- 画面遷移は「空き状況 → 入力フォーム → 完了」および「管理画面」の2系統。

## 2. 現状の主要機能
### 利用者向け
- カレンダー表示（7日分、6:00〜12:00）で予約可否を表示（◎/×）。
- 予約送信（名前・電話番号・出発地・行き先 + URLクエリから日時）。

### 管理者向け
- 簡易ログイン後に予約一覧表示。
- 予約削除。

### API
- 初期データ取得: `/api/getInitData`
- メニュー取得: `/api/menu`
- 予約一覧: `/api/getReservations`
- 予約作成: `/api/createReservation`
- 予約削除: `/api/deleteReservation`

## 3. リスク/課題（優先度順）
1. **機密情報・認証の脆弱性**
   - 管理画面パスワードがクライアントJSに平文固定（`1234`）。
   - APIに認証・認可がなく、`/api/deleteReservation` などが公開状態。
2. **環境ごとのURL設定不整合**
   - `app.js` / `form.html` は `YOUR-WORKER-URL` のプレースホルダ。
   - `admin.js` は本番らしき固定URLを直書き。
3. **予約導線の欠落**
   - `index.html` 上の空き枠がクリック不可（表示のみ）。
   - `form.html` は `date` / `time` クエリ依存で、リンク元実装が見当たらない。
4. **例外処理とUX**
   - フロントは送信失敗時でも `thanks.html` へ遷移する実装がある。
   - APIエラー時の詳細ハンドリングが薄い。
5. **時刻/タイムゾーン整合性**
   - `new Date()` や文字列結合で日時処理しており、実行環境TZ差異の影響余地。
6. **データ整合性/同時実行性**
   - 重複チェック後にINSERTしており、競合時の完全な排他はDB制約依存。

## 4. 良い点
- APIエンドポイントが機能別に整理され、読みやすい。
- 予約作成時にブロック枠を自動生成し、削除時に対応解除している。
- `app.js` 側でAPI失敗時に空データで描画継続する配慮がある。

## 5. 改善ロードマップ（短期）
1. 管理画面/削除APIに認証導入（最低限トークン検証、理想はJWT+期限）。
2. APIベースURLを単一設定化（環境変数 or 1ファイル定数管理）。
3. カレンダー枠クリックで `form.html?date=...&time=...` へ遷移実装。
4. 予約POSTレスポンスを検証し、失敗時はエラー表示・遷移停止。
5. DBに一意制約（`blocks(date, time)`）を付与して二重予約を防止。

## 6. 追加で確認したい事項
- 実運用時のタイムゾーン要件（JST固定か、拠点ごと可変か）。
- 管理者権限の要件（閲覧のみ、削除可、価格編集可など）。
- 料金計算ロジック（現状 `total_price = 0` 固定）をどこで確定させるか。
